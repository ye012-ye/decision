# Decision Web Frontend Design

> 日期：2026-04-09
> 范围：为 `decision` 项目新增独立前端工程 `decision-web`
> 目标：提供可联调的智能客服工作台，接入聊天流、知识库管理、工单管理

## 1. 概述

当前仓库包含两个后端模块：

- `decision-app`：主业务应用，提供聊天 SSE、知识库管理、工单业务能力
- `decision-mcp-server`：独立 MCP 数据库工具服务

本次设计新增一个独立前端工程 `decision-web`，作为与 `decision-app` 联调的真实操作界面。它不是项目官网，也不是传统后台模板，而是一个围绕“AI 对话驱动业务处理”的智能客服决策台。

第一版必须满足以下目标：

- 能对接现有 `POST /api/chat/stream` SSE 流式聊天接口
- 能对接现有知识库接口 `GET/POST/PUT/DELETE /api/kb/**`
- 新增工单 REST 接口后，前端可完成工单创建、查询、状态流转、关闭
- 页面视觉具备明确产品感，不使用通用后台 UI 套壳
- 桌面端优先，移动端至少可用

不在本次范围内：

- 完整权限系统
- 登录鉴权
- 多角色工作流
- 大屏 BI 和复杂报表
- 独立客服坐席排班系统

## 2. 目标用户与核心场景

### 2.1 目标用户

- 客服人员：通过 AI 辅助完成客户答复、问题诊断、工单处理
- 运营/支持人员：查看知识库、工单状态、处理日志
- 项目演示对象：快速理解该系统的“对话 + 工具 + 工单”闭环能力

### 2.2 核心场景

1. 用户在工作台发起会话，实时查看 AI 的推理和工具调用过程
2. AI 根据用户诉求生成处理建议、知识命中结果或工单结果
3. 坐席从右侧业务面板查看工单详情，必要时手动创建或更新工单
4. 坐席进入知识库页面创建知识库、上传文档、查看摄入状态
5. 坐席进入工单页面检索工单、查看日志、更新状态、关闭工单

## 3. 技术方案

### 3.1 工程结构

在仓库根目录新增平级模块：

```text
decision-web/
```

与现有模块关系如下：

```text
decision/
├─ decision-app/
├─ decision-mcp-server/
├─ decision-web/
└─ docs/
```

### 3.2 技术选型

- 框架：Vue 3
- 构建工具：Vite
- 语言：TypeScript
- 路由：Vue Router
- 状态管理：Pinia
- 样式：原生 CSS + CSS Variables
- 网络请求：Fetch API 封装
- SSE：原生 `fetch + ReadableStream` 方式消费 `text/event-stream`

不引入重量级后台 UI 组件库，原因如下：

- 本项目强调视觉辨识度，通用后台库会明显拉低成品感
- 核心页面是工作台而不是表格管理系统，布局和交互应按业务定制
- 第一版组件规模可控，自建基础组件成本合理

### 3.3 前后端联调方式

- 前端开发服务：`http://localhost:5173`
- 后端接口服务：`http://localhost:8080`
- 使用 Vite 代理 `/api` 到 Spring Boot，避免开发期跨域与 Cookie 问题

生产模式可选两种：

1. 前端独立部署，Nginx 反向代理后端
2. 前端构建产物由 `decision-app` 静态托管

本次设计不绑定最终部署方式，只保证两种模式均可支持。

## 4. 产品形态与信息架构

### 4.1 一级路由

保留三个一级页面：

- `/workspace`：智能客服工作台
- `/knowledge`：知识库管理
- `/tickets`：工单管理

### 4.2 导航结构

全局采用左侧竖向导航，保持工作台应用的稳定感：

- 顶部：产品名 `Decision Console`
- 中部：`工作台 / 知识库 / 工单`
- 底部：当前环境、接口状态、构建版本

### 4.3 页面职责

#### `/workspace`

工作台是系统主场景，承载 AI 会话和业务联动。

布局采用三栏结构：

- 左栏：会话入口、最近会话、快捷动作
- 中栏：聊天舞台，展示消息流和 ReAct 过程
- 右栏：上下文面板，展示客户、工单、知识命中、工具结果

#### `/knowledge`

知识库页面用于管理知识库与文档，不做复杂资产平台，只提供本项目所需的核心能力：

- 知识库列表
- 创建/编辑知识库
- 文档上传
- 文档列表
- 摄入状态查看

#### `/tickets`

工单页面用于手动管理与追踪工单：

- 条件筛选
- 列表浏览
- 查看详情
- 查看操作日志
- 更新状态
- 关闭工单

## 5. 视觉系统

### 5.1 视觉方向

视觉主题定义为：

> 深色智能调度台，克制、专业、可操作，突出“AI 在业务场景中协同执行”的中控感。

不采用传统浅色后台和满屏卡片。整体强调：

- 深层背景
- 精细分隔
- 稳定排版
- 少量高亮
- 局部玻璃雾面

### 5.2 色彩

建议使用以下色系：

- 页面主背景：深墨蓝、石墨灰
- 主要文字：高亮冷白
- 次要文字：低饱和灰蓝
- 主强调色：琥珀橙
- 信息状态色：蓝绿色
- 风险状态色：红橙色

主强调色仅用于：

- 当前导航项
- 主按钮
- 时间线高亮节点
- 关键状态标签

### 5.3 字体

- 中文主字体：`Noto Sans SC`
- 英文与数字强调字体：`Space Grotesk`

使用规则：

- 品牌名和关键数字使用英文字体
- 主体信息和操作文本使用中文字体
- 避免同时引入第三种字体，保持克制

### 5.4 背景与质感

- 页面背景使用暗色渐层，不是纯黑
- 局部叠加轻微网格纹理
- 工作台主舞台后方加入低透明径向光斑
- 面板边界依赖细线和透明度差，而不是厚重边框

### 5.5 组件原则

- 默认不使用卡片堆砌
- 以分栏、列表、抽屉、时间线组织信息
- 只有在交互边界明确时才使用面板容器
- 阴影保持极轻，靠对比与层次建立信息结构

## 6. 页面详细设计

### 6.1 工作台 `/workspace`

#### 6.1.1 左栏

内容包括：

- 新建会话按钮
- 最近会话列表
- 客户快捷入口
- 快速模板入口，例如“投诉处理”“物流咨询”“工单查询”

行为要求：

- 当前会话高亮
- 新建会话后自动生成本地会话标识
- 会话切换不刷新页面

#### 6.1.2 中栏聊天舞台

内容包括：

- 顶部会话标题区
- 消息主流
- ReAct 时间线区
- 底部输入区

消息类型视觉区分：

- 用户消息：靠右，视觉更轻
- AI 普通回复：靠左，作为主内容
- Thought：低对比辅助文本
- Action：高辨识工具胶囊 + 参数摘要
- Observation：结构化结果块
- Answer：最终结论，视觉最明确

底部输入区包括：

- 文本输入
- 快捷发送按钮
- 中断/重试入口

#### 6.1.3 右栏上下文面板

采用顶部标签切换：

- 客户
- 工单
- 工具结果
- 知识命中

联动原则：

- 当最新消息涉及工单时，自动切到“工单”
- 当最新消息涉及知识检索时，自动切到“知识命中”
- 用户可手动切换并保持当前上下文

#### 6.1.4 工作台中的工单入口

右栏工单区需支持：

- 展示当前工单编号、类型、优先级、状态
- 查看处理人、处理组、创建时间、处理结果
- 快速更新状态
- 快速关闭工单
- 手动新建工单

### 6.2 知识库 `/knowledge`

页面划分为左右结构：

- 左侧：知识库列表
- 右侧：选中知识库的详情和文档列表

核心操作：

- 创建知识库
- 编辑知识库
- 删除知识库
- 上传文档
- 查看文档摄入状态
- 删除文档

上传区需要清晰展示：

- 支持格式
- 当前上传状态
- 成功/失败结果
- 文档摄入中的轮询刷新

### 6.3 工单 `/tickets`

页面布局：

- 顶部筛选栏
- 中部工单列表
- 右侧详情抽屉或固定详情区

筛选条件：

- 工单编号
- 客户 ID
- 状态
- 类型
- 优先级

列表项展示：

- 工单编号
- 标题
- 类型
- 优先级
- 状态
- 处理人
- 创建时间

详情区展示：

- 工单基础信息
- 处理结果
- 操作日志时间线
- 状态更新操作
- 关闭工单操作

## 7. 交互设计

### 7.1 动效原则

动效要有存在感，但不能干扰操作。只保留三类关键动画：

- 页面首屏渐入
- ReAct 时间线逐条滑入
- 右侧上下文面板平滑切换

### 7.2 工作台关键交互

- 发送消息后，输入区立即进入处理中状态
- SSE 返回内容按事件流逐步落位
- 工具调用节点出现时，时间线节点短暂高亮
- 最终答案出现后，相关业务面板同步刷新
- SSE 结束时，输入区恢复可编辑

### 7.3 工单交互

- 工单状态更新在局部完成，不整页刷新
- 关闭工单前要求二次确认
- 更新成功后，详情区与日志区同步刷新

### 7.4 知识库交互

- 上传多个文件时显示单文件状态
- 处于摄入中的文档显示处理中标识
- 失败状态需提供明确原因或至少暴露后端返回信息

## 8. 数据流设计

### 8.1 聊天流

1. 用户在工作台输入消息
2. 前端发起 `POST /api/chat/stream`
3. 前端解析 SSE 块并转为内部事件对象
4. 事件追加到当前会话消息流
5. 根据事件内容刷新右侧上下文
6. 收到 `done` 后结束本轮请求
7. 收到 `error` 后保留已生成内容并提示失败

### 8.2 知识库流

1. 进入页面加载知识库列表
2. 选择知识库后加载详情与文档列表
3. 上传文档成功后将新文档插入列表
4. 对摄入中的文档轮询状态接口
5. 状态进入完成或失败后停止轮询

### 8.3 工单流

1. 页面加载或筛选时请求工单列表
2. 选中工单后请求详情和日志
3. 更新状态后局部刷新详情与日志
4. 关闭工单后回写状态为 `CLOSED`
5. 工作台侧栏与工单页共用工单查询模型

## 9. 前端状态模型

建议拆分为三个主 store：

- `workspaceStore`
- `knowledgeStore`
- `ticketStore`

### 9.1 `workspaceStore`

负责：

- 当前会话列表
- 当前会话消息流
- 当前 SSE 连接状态
- 当前右侧上下文数据

### 9.2 `knowledgeStore`

负责：

- 知识库列表
- 当前知识库详情
- 文档列表
- 上传状态
- 摄入轮询状态

### 9.3 `ticketStore`

负责：

- 工单筛选条件
- 工单列表
- 当前工单详情
- 当前工单日志
- 工单操作提交状态

- `POST /api/chat/stream`

- `GET /api/kb/{kbCode}`

- `GET /api/kb/{kbCode}/documents`
- `GET /api/kb/{kbCode}/documents/{docId}/status`

### 10.2 新增工单接口

本次需要在 `decision-app` 中新增独立 REST API，供前端调用，不复用 AI 工具入参 `WorkOrderReq`。

建议接口如下：

- `GET /api/work-orders`
- `POST /api/work-orders`
- `GET /api/work-orders/{orderNo}`
- `PATCH /api/work-orders/{orderNo}/status`
- `POST /api/work-orders/{orderNo}/close`
- `GET /api/work-orders/{orderNo}/logs`

- `WorkOrderCreateReq`
- `WorkOrderStatusUpdateReq`

- `priority`
- `title`
- `description`
- `customerId`
- `sessionId`

#### `WorkOrderStatusUpdateReq`

- `status`
- `note`
- `operator`

#### `WorkOrderCloseReq`

- `resolution`
- `operator`

#### `WorkOrderVO`

- `orderNo`

- `priority`
- `status`

- `description`
- `customerId`

## 12. 异常处理

### 12.1 聊天异常

- SSE 中断时保留已接收内容
- 页面展示“连接中断，可重试”的明确提示
- 避免直接清空当前消息流
- 支持用户重新发送上一条消息

### 12.2 工单异常

- 创建失败时展示后端 `msg`
- 状态更新失败时不提前修改本地状态
- 详情刷新失败时保留旧数据并提示局部失败

### 12.3 知识库异常

- 上传失败时展示单文件失败
- 状态轮询失败时停止当前轮询并给出提示
- 空列表、空搜索结果、空知识库状态都使用专门空态界面

### 12.4 接口兼容

前端统一适配当前后端返回：

- 常规 REST：`Result<T> = { code, msg, data }`
- SSE：`thought / action / observation / answer / done / error`

## 13. 响应式策略

桌面端是主目标，推荐设计宽度 `1440px` 左右优化。

在窄屏下：

- 左侧导航收折
- 工作台右侧上下文栏改为抽屉
- 工单详情改为全屏抽屉
- 知识库页面从双栏退化为上下结构

不要求第一版移动端具备完整桌面效率，但必须可访问、可操作、可阅读。

## 14. 实现边界与取舍

第一版优先保证：

- 主工作台联调真实可用
- 工单 CRUD 主链路可用
- 知识库管理主链路可用
- 视觉成品明显优于通用管理台模板

有意不做：

- 复杂富文本编辑
- 多标签会话持久化到服务端
- 拖拽编排
- 仪表盘统计模块
- MCP 服务直连浏览器

## 15. 验证与测试要求

### 15.1 联调验证

至少验证以下三条主链路：

1. 聊天工作台可正确消费 SSE 事件并渲染四类过程节点
2. 工单可创建、查看详情、更新状态、关闭
3. 知识库可创建、上传文档、查看摄入状态

### 15.2 前端测试

建议覆盖：

- API 封装单测
- SSE 事件解析单测
- 关键页面组件渲染测试
- 至少一个端到端联调流程

### 15.3 体验验证

上线前至少人工检查：

- 桌面端布局完整
- 深色主题对比度足够
- 输入、加载、失败、空状态完整
- 工单和知识库页面不会因为局部失败导致整页崩溃

## 16. 结论

本方案将 `decision` 项目补齐为“三层结构”：

- 独立前端 `decision-web`
- 主业务应用 `decision-app`
- MCP 工具服务 `decision-mcp-server`

前端以“智能客服决策台”为核心形态，围绕聊天、知识、工单三条主线组织功能。它不是后台模板，也不是展示页，而是一个能够直接演示并支撑真实业务联调的操作界面。

本设计确认后，下一步应进入实现计划阶段，拆分为：

1. 脚手架与基础样式
2. 工作台页面与 SSE 聊天联调
3. 工单 REST API 与前端工单页
4. 知识库页面联调
5. 测试与收尾
